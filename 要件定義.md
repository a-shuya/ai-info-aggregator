## 個人用情報収集サイト開発プロジェクト 要件定義書

### 1. 概要

#### 1.1. プロジェクトの背景
現代では、ビジネスやテクノロジーに関する情報が複数のWebサイトに分散しており、関心のある情報を網羅的に収集するには多くの時間と手間を要する。この非効率を解消するため、指定された複数のWebサイトから最新記事を自動的に集約し、一元的に閲覧できる個人用の情報収集サイトを構築する。

#### 1.2. プロジェクトの目的
本プロジェクトの目的は、指定されたWebサイト群から最新3日分（本日、昨日、一昨日）の記事情報を効率的に収集・整理し、快適な閲覧体験を提供するWebアプリケーションを、**私的利用**かつ**無料**で開発・運用することである。

### 2. システムの利用者と利用シーン

*   **利用者:** 開発者本人
*   **利用シーン:**
    *   毎朝、出勤前や業務開始時に、業界の最新動向をまとめてチェックする。
    *   移動中や休憩時間などに、スマートフォンで手軽に情報を確認する。

### 3. システムの全体像

本システムは、定期的に対象サイトから記事情報を取得する「**データ収集部（バックエンド）**」と、収集した情報をWebページとして表示する「**データ表示部（フロントエンド）**」の2つの主要コンポーネントで構成される。

*   **データ収集部:** スケジュールに基づき、各サイトを巡回（スクレイピング）し、記事の「タイトル」「URL」「公開日」「サイト名」などを抽出し、データベースまたはファイルに保存する。
*   **データ表示部:** 保存されたデータを読み込み、「本日」「昨日」「一昨日」のカテゴリに分類して、Webページ上に一覧表示する。

### 4. 機能要件

#### 4.1. データ収集機能 (F-01)
| ID | 機能名 | 機能概要 |
| :--- | :--- | :--- |
| F-01-01 | 記事情報収集 | 指定された全サイトのHTMLを解析（スクレイピング）し、個々の記事情報を抽出する。 |
| F-01-02 | 抽出項目 | 少なくとも以下の情報を抽出・取得する。<br>・記事タイトル<br>・元記事のURL<br>・公開日時<br>・サイト名<br>・（可能であれば）サムネイル画像のURL |
| F-01-03 | データ保存 | 抽出した記事情報を、後続の処理で利用しやすい形式（JSONファイル、SQLiteデータベース等）で保存する。重複する記事は保存しない。 |
| F-01-04 | 定期実行 | 収集処理を定期的（例: 1時間に1回）に自動で実行する。 |

#### 4.2. データ表示機能 (F-02)
| ID | 機能名 | 機能概要 |
| :--- | :--- | :--- |
| F-02-01 | 日付別一覧表示 | 収集した記事を「本日の記事」「昨日の記事」「一昨日の記事」のセクションに分けて表示する。 |
| F-02-02 | 記事リスト表示 | 各セクション内で、サイトごと、または時系列で記事を整理して表示する。各記事には「サイト名」と「記事タイトル」を表示する。 |
| F-02-03 | 外部リンク | 記事タイトルをクリックすると、ブラウザの新しいタブで元記事のページが開く。 |
| F-02-04 | レスポンシブ対応 | PC、タブレット、スマートフォンの各デバイスで最適に表示されるレスポンシブデザインを採用する。 |

### 5. 非機能要件

#### 5.1. 性能
*   **ページ表示速度:** ユーザーがストレスを感じないよう、3秒以内にページコンテンツが表示されることを目指す。
*   **データ収集時間:** 全サイトの収集処理が、次回の実行スケジュールに影響しない時間内（例: 15分以内）に完了すること。

#### 5.2. 可用性
*   私的利用のため、24時間365日の常時稼働は必須としない。ただし、開発者が利用したい時間帯には安定してアクセスできること。

#### 5.3. 運用・保守性
*   **設定の分離:** 収集対象サイトのURLやスクレイピングのルール（CSSセレクタ等）は、コード本体から分離された設定ファイルで管理し、サイト構造の変更に迅速に対応できるようにする。
*   **エラーハンドリング:** 特定のサイトで収集エラーが発生しても、システム全体が停止せず、他のサイトの収集は継続されるようにする。エラー発生時にはログを出力する。

#### 5.4. セキュリティ
*   本システムは個人利用を前提とし、ユーザー認証や個人情報の入力機能は設けないため、高度なセキュリティ対策は不要とする。
*   ただし、Webサーバーを外部に公開する場合は、基本的なセキュリティプラクティス（不要なポートを閉じる等）を遵守する。

#### 5.5. 遵法性・倫理（最重要項目）
*   **利用規約の遵守:** スクレイピングを行う前に、各対象サイトの利用規約を確認し、情報収集が許可されているかを確認する。
*   **サーバー負荷への配慮:** 対象サイトのサーバーに過度な負荷をかけないよう、リクエスト間隔を適切に空ける（例: 1リクエストごとに1〜3秒の待機時間を設ける）。
*   **robots.txtの尊重:** 各サイトの`robots.txt`ファイルを確認し、クローラーのアクセスが禁止されているページにはアクセスしない。
*   **私的利用の徹底:** 収集したデータの利用は、個人による情報収集の範囲に限定する。第三者への再配布、公開、販売、および営利目的での利用は一切行わない。

### 6. システム構成案（無料での実現を前提）

無料で開発・運用を行うため、各種クラウドサービスの無料枠を最大限に活用する構成を提案する。

*   **データ収集（バックエンド）**
    *   **言語:** Python
    *   **ライブラリ:** `requests`（HTTP通信）、`Beautiful Soup 4`（HTML解析）、`Playwright`（JavaScriptで描画されるサイト向け）
    *   **実行環境:** **GitHub Actions**。無料で定期的なスクリプト実行（Cronジョブ）が可能。

*   **データ保存**
    *   **形式:** **JSONファイル** または **SQLite**。
    *   **保存場所:** GitHubリポジトリ内。GitHub Actionsで生成したファイルをリポジトリにコミットして保存する。

*   **データ表示（フロントエンド）**
    *   **フレームワーク:** **Next.js** (React) または **Astro**。静的サイト生成（SSG）に優れ、高速なサイトを構築できる。
    *   **ホスティング:** **Vercel** または **Netlify**。GitHubリポジトリと連携し、無料で自動デプロイが可能。

#### 構成フロー
1.  **GitHub Actions**が毎日数回、定刻にPythonスクリプトを実行。
2.  Pythonスクリプトが対象サイトをスクレイピングし、記事データを**JSONファイル**として生成。
3.  生成されたJSONファイルをリポジトリに自動でコミット＆プッシュ。
4.  リポジトリへのプッシュをトリガーに、**Vercel**が自動でビルド処理を開始。
5.  **Next.js/Astro**がJSONファイルを読み込み、静的なHTMLページを生成してデプロイ。
6.  ユーザーはVercelのURLにアクセスし、最新の記事一覧を閲覧する。

### 7. 開発スコープ

#### 7.1. 対象範囲 (In Scope)
*   本要件定義書に記載された機能要件（F-01, F-02）の実装。
*   指定された全19サイト（日経ビジネスx1, Business Insiderx4, 日経xTECHx1, ITmediax1, はてなブックマークx2, Zennx8, Google Cloudx3）のスクレイピング処理の実装。

#### 7.2. 対象範囲外 (Out of Scope)
*   ユーザー認証、ログイン機能
*   記事のキーワード検索機能
*   記事の「お気に入り」「既読」管理機能
*   商用利用を目的とした機能全般
*   本要件定義書に記載のないサイトの追加（将来的な拡張は考慮する）
